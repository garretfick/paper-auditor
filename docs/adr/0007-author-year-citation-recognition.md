# Author-year Citation recognition: broaden, filter, resolve

The audit recognizes author-year **Citations** (`(Smith, 2020)`, `Smith et al. (2019)`) — not only Pandoc `[@key]` — through a three-stage pipeline: a high-recall regex emits **candidates**, an optional LLM **Citation Filter** discards the citation-shaped false positives, and the **Resolver** pairs the survivors to the **Bibliography** by first-author surname + year. The reason a future reader might assume the opposite — "the Resolver only understands `[@key]`, so author-year prose is simply unsupported" — is that this is exactly the silent asymmetry issue #37 documented: the LLM **Claim Extractor** read author-year prose fine, but the regex Resolver saw nothing, so a **Paper** full of author-year **Citations** drew zero `UnresolvedCitation` **Findings** and looked clean when it was not. The pipeline closes that gap: the candidate regex now sees author-year forms, so they reach resolution instead of vanishing.

The three stages are deliberately separated by responsibility, and one constraint governs the split: **the Bibliography is never sent to a model, and the model never normalizes a Citation Key.** The reason a future reader might assume the opposite — "just give the LLM the `.bib` and let it map `(Smith, 2020)` to `smith2020`" — is that it looks like the shortest path. It is rejected because citation↔**Bibliography** pairing must stay deterministic and auditable: surname + year matching against parsed **Bibliography** entries (see `matchAuthorYear`) is reproducible and inspectable, whereas an LLM key-normalization step would make resolution depend on model judgement and hide mismatches inside a prompt. The LLM's role is strictly narrower — a discriminator answering "is this candidate a real **Citation**?" — and it sees only candidate snippets and their surrounding context, never the reference list.

Stage 1 (broaden) is high-recall on purpose. The regex over-matches — `(Table 2020)`, `Section 3 (2024)`, and pandoc-crossref refs like `[@fig:1]` all surface as candidates — because a recall-first matcher with a precision filter behind it beats a clever single regex that silently drops the long tail of real-world citation styles. Stage 2 (the **Citation Filter**) is where precision is recovered, and it is optional: with no Filter wired, every candidate flows straight to resolution. Trade-off accepted: **running without a Filter trades precision for recall** — false-positive candidates whose surname+year happens not to match the **Bibliography** will raise spurious `UnresolvedCitation` **Findings**, and the local-model Filter (`llama3.1:8b`, per ADR-0004) is itself a noise source, the same tax ADR-0002 already accepts for the **Claim Extractor**. The `Confidence` field and prompt tuning are the mitigations, not a stricter regex.

Known limitations, recorded so they read as deliberate rather than as bugs: surname+year resolution matches on the **first** author only, so a **Citation** that names a later author resolves only if the first author and year also line up; multi-citation parentheticals (`(Smith 2020; Jones 2021)`) are not yet split into separate candidates; and a genuine author-year **Citation** whose **Source** is missing from the **Bibliography** is correctly surfaced as `UnresolvedCitation` — that is the **Finding** the audit exists to produce, not a false positive.
