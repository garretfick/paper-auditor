# Partial-workflow CI/CD with `justfile` as the deep module

Both CI and local development invoke `just build` to validate a change. `just build` is one recipe — it runs `setup`, then typecheck (`tsc -b`), then the vitest suite, then ESLint, then Prettier — in that order, in a single recipe. The reason a future reader might assume the opposite — splitting build, test, and lint across three top-level recipes — is that doing so would let a packaging matrix in the deployment workflow start in parallel with tests still running (whichever finished first). The deployment workflow's invariant is **no artifact unless every check passed**, so build, test, and lint collapse into one recipe invoked by one `partial_build.yaml`. The downstream packaging job declares `needs: [release, build]` and the wiring then guarantees the invariant without anyone needing to read the partial to verify it.

GitHub Actions YAML files contain only orchestration — checkout, install `just`, install pnpm via corepack, run a single `just` recipe, upload artifacts. No shell logic, no inline `bash -c`. If a step grows logic, the logic moves to the `justfile`. The `justfile` is therefore the deep module: a narrow public surface (`setup`, `build`, and later `get-next-version`, `version`, `commit-version`, `package`) hiding all multi-tool, multi-package coordination behind it. Workflow YAML stays one-screen. The same recipe runs locally and in CI so "works on my laptop" and "works in CI" cannot diverge through silent drift in YAML.

Deployment secrets live in a `production` GitHub Environment that only deployment-side partials enter. PR-triggered runs never enter `production`, so a malicious PR cannot reach a publish token. The version-bump partial (`partial_version.yaml`) is gated by that environment and is the sole writer of version commits and `v*` tags via a bot PAT scoped to push on this repo. The integration partial (`partial_build.yaml`) runs without environment scope and has no path to the publish token. Trade-off accepted: the `production` environment must be created manually in the repo's settings and its restriction to the `main` branch enforced by deployment branch rules — there is no Terraform or repo-as-code provisioning for it in v1. Reference prior art: [ironplc/ironplc](https://github.com/ironplc/ironplc), whose partial-workflow shape this repo follows; the version recipe is simplified to a fixed `minor` bump for v1.
